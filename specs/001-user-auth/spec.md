## User Scenarios & Testing _(mandatory)_

### User Story 1 - 邮箱密码注册并使用（优先级 P1）

新访客通过邮箱与密码创建账号，收到一封验证邮件，点击链接后邮箱被标记为已验证，随即可以登录、聊天。

**为什么是 P1**：这是 Stage 1 公开产品最基础的注册路径；如果邮箱密码注册走不通，第三方登录只是少数用户的备选方案，平台依然无法规模化获客。

**独立测试**：可以用一个测试邮箱走完"注册 → 收件箱点链接 → 登录 → 发一条消息 → 退出 → 重新登录 → 看到历史消息"全流程，无需配置任何第三方 OAuth 凭据。

**验收场景**：

1. **Given** 访客未登录，**When** 提交邮箱 + 密码，**Then** 系统创建账号、发送验证邮件、跳转到"请查收邮件"提示页。
2. **Given** 用户收到验证邮件，**When** 点击邮件中的链接，**Then** 邮箱被标记为已验证，用户自动登录或被引导到登录页。
3. **Given** 邮箱已验证，**When** 用同一邮箱密码登录，**Then** 登录成功，进入主界面。
4. **Given** 邮箱未验证，**When** 用同一邮箱密码尝试登录，**Then** 登录被拒绝，提示"请先验证邮箱"。
5. **Given** 验证邮件已超过有效期（默认 24 小时），**When** 点击链接，**Then** 提示链接已过期，并提供"重新发送验证邮件"入口。

### User Story 2 - GitHub 一键登录（优先级 P2）

访客点击 "Continue with GitHub" 按钮，跳转到 GitHub 授权页，回调后创建或关联账号并直接登录。

**为什么是 P2**：减少注册阻力，对开发者群体尤其重要；但缺少它用户仍可通过邮箱密码完成注册。

**独立测试**：配置 GitHub OAuth 凭据后，访客点击按钮、走完 GitHub 授权、回到应用并能立即发消息——全流程无需离开浏览器。

**验收场景**：

1. **Given** 访客未登录，**When** 点击 "Continue with GitHub"，**Then** 跳转到 GitHub 授权页。
2. **Given** GitHub 授权成功，**When** 回调到应用，**Then** 创建账号（首次）或匹配已有账号（邮箱相同），并直接登录。
3. **Given** GitHub 授权失败或被拒绝，**When** 回调到应用，**Then** 显示错误提示，用户留在登录页。

---

### User Story 3 - Google 一键登录（优先级 P3）

访客点击 "Continue with Google" 按钮，跳转到 Google 授权页，回调后创建或关联账号并直接登录。

**为什么是 P3**：覆盖非开发者用户；但与 GitHub OAuth 是对称功能，缺失时大部分用户仍可走邮箱密码。

**独立测试**：配置 Google OAuth 凭据后，访客点击按钮、走完 Google 授权、回到应用并能立即发消息——全流程无需离开浏览器。

**验收场景**：

1. **Given** 访客未登录，**When** 点击 "Continue with Google"，**Then** 跳转到 Google 授权页。
2. **Given** Google 授权成功，**When** 回调到应用，**Then** 创建账号（首次）或匹配已有账号（邮箱相同），并直接登录。
3. **Given** Google 授权失败或被拒绝，**When** 回调到应用，**Then** 显示错误提示，用户留在登录页。

---

### User Story 4 - 聊天会话归属与隔离（横切关注点，所有 P1/P2/P3 都需要）

每个登录用户只能看到、打开、修改、删除自己的聊天会话。

**为什么单独提**：登录的本质目标就是数据隔离——任何登录方式都必须满足这一点；如果只满足登录 UI 而不满足隔离，等于没做。

**独立测试**：注册 A、B 两个账号；A 创建 thread 并发消息；B 在登录态下列出 thread 应当为空，访问 A 的 thread URL 应被拒绝。

**验收场景**：

1. **Given** 用户已登录，**When** 列出聊天会话，**Then** 只看到自己创建的会话。
2. **Given** 用户已登录，**When** 通过 URL 直接访问他人 thread，**Then** 返回"无权访问"或 404，不泄漏对方数据。
3. **Given** 用户已登录，**When** 删除自己的 thread，**Then** thread 立即从列表消失；他人不受影响。
4. **Given** 用户被删除，**When** 系统清理，**Then** 该用户所有 thread 一并删除。

---

### Edge Cases

- **重复邮箱注册**：邮箱已存在（不论注册方式）时，再次用邮箱密码注册必须返回"该邮箱已被注册"——不泄漏账号是否存在。
- **OAuth 邮箱与已有邮箱密码账号冲突**：GitHub/Google 返回的邮箱与已有邮箱密码账号相同时，应当登录到已有账号（合并第三方身份）而非创建新账号。
- **密码强度不足**：注册时强制最低长度（默认 8 字符），不满足时立即在前端提示，不发送验证邮件。
- **邮件格式不合法**：邮箱格式不合规时立即拒绝，不进入"已发送"状态。
- **未登录访问受保护页面**：用户未登录直接访问主页，应当被重定向到登录页（带 returnTo）。
- **登出后再访问受保护资源**：会话失效后访问任意受保护 API 必须返回 401。
- **第三方 OAuth 服务不可用**：回调失败时不应创建孤立账号；提示用户稍后重试。
- **第三方邮件服务不可用**：用户提交注册时邮件投递失败，应当提示"暂时无法发送验证邮件，请稍后重试"，并保留已创建账号待用户重试发送。
- **删除账号时的并发**：用户在删除自己账号时正在另一个标签页发消息——正在进行的会话必须立即失效；新消息被拒绝。
- **验证邮件重新发送**：未验证账号登录被拒时，可点"重新发送验证邮件"；每次发送都使旧的链接失效。

## Requirements _(mandatory)_

### Functional Requirements

#### 注册与登录（邮箱密码）

- **FR-001**：系统必须允许用户通过邮箱 + 密码注册账号。
- **FR-002**：系统必须对注册请求中的密码强制以下规则：长度不少于 8 字符，且必须同时包含字母与数字（字母大小写均可）。
- **FR-003**：系统必须对邮箱格式做合法性校验，非法邮箱立即拒绝。
- **FR-004**：系统必须在注册成功后立即通过第三方邮件服务发送验证邮件。
- **FR-005**：验证邮件必须包含一个一次性、限时（默认 24 小时内有效）的链接；邮件正文必须采用响应式 HTML 模板，在主流邮件客户端（Gmail / Outlook / Apple Mail）下视觉一致。
- **FR-006**：系统必须要求邮箱密码用户在邮箱验证通过后才能登录。
- **FR-007**：系统必须在用户登录被拒（邮箱未验证）时提供"重新发送验证邮件"的入口。
- **FR-008**：重新发送验证邮件必须使旧链接失效。

#### 第三方登录（OAuth）

- **FR-009**：系统必须支持 GitHub 第三方登录。
- **FR-010**：系统必须支持 Google 第三方登录。
- **FR-011**：OAuth 登录成功后，必须根据邮箱匹配已有账号；邮箱不存在则创建新账号。
- **FR-012**：OAuth 用户的邮箱默认视为已验证（由 provider 保证）。
- **FR-013**：OAuth 授权失败或被拒绝时，必须在登录页提示错误，不创建账号。

#### 会话

- **FR-014**：系统必须为登录用户维护持久会话，跨页面刷新、跨浏览器关闭均保持登录态；会话有效期固定为 7 天，到期后需重新登录。
- **FR-015**：系统必须支持用户主动登出，登出后立即失效当前会话。
- **FR-016**：未登录或会话失效的请求访问受保护资源，必须返回 401 或重定向到登录页。

#### 聊天会话归属（横切）

- **FR-017**：每个聊天会话必须归属于一个用户，不允许存在无主会话。
- **FR-018**：列出聊天会话的接口必须只返回当前登录用户拥有的会话。
- **FR-019**：读取、更新、删除单个聊天会话的接口必须验证当前用户是会话所有者；否则返回 401/403/404（不可泄漏存在性）。
- **FR-020**：创建新聊天会话时必须自动绑定到当前登录用户。
- **FR-021**：删除用户账号时必须级联删除其所有聊天会话。

#### 错误与异常

- **FR-022**：重复邮箱注册必须被拒绝（邮箱密码路径），错误信息不泄漏账号是否存在。
- **FR-023**：第三方邮件服务不可达时，必须将注册请求视为失败（不创建未验证账号）并提示用户稍后重试。
- **FR-024**：OAuth 提供方不可达时，登录页必须提示用户，不静默失败。
- **FR-025**：当邮件服务商返回每日配额超限（HTTP 429 / provider 配额错误）时，系统必须拒绝当前的注册请求（不创建账号），并向用户明确提示"邮件暂时无法发送，请稍后重试"。

### Key Entities _(include if feature involves data)_

- **用户（User）**：系统的核心身份实体；属性包括唯一标识符、邮箱（唯一）、邮箱是否已验证、显示名、头像、创建时间；至少一种登录方式（密码或第三方）。
- **第三方账号（OAuth Account）**：表示用户与第三方身份提供方（GitHub / Google）的绑定关系；包含第三方用户标识、绑定的本地用户标识、提供方类型。
- **会话（Session）**：表示一次登录态；归属于用户；有过期时间；登出或过期即失效。
- **验证令牌（Verification Token）**：表示一次邮箱验证的待办；包含目标邮箱、唯一令牌、过期时间；使用或重新发送时失效。
- **聊天会话（Thread）**：已有的实体；本期新增"归属用户"的强制属性，不再允许无主。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**：新用户从打开首页到发出第一条消息的端到端时间不超过 2 分钟（不计邮件投递延迟，假设邮件在 60 秒内送达）。
- **SC-002**：99% 的验证邮件在 60 秒内送达收件箱（由邮件服务商 SLA 衡量）。
- **SC-003**：任意登录用户都无法通过 UI 或 API 看到、修改、删除他人聊天会话——通过权限测试验证 100% 隔离。
- **SC-004**：OAuth（GitHub/Google）从点击按钮到进入主界面的端到端时间不超过 10 秒（不计用户在 provider 页停留时间）。
- **SC-005**：用户主动登出后，立即无法访问任何受保护资源；下一个 API 请求必须返回 401。
- **SC-006**：验证链接超过 24 小时后自动失效；过期链接点击必须被拒绝。
- **SC-007**：删除用户后，该用户所有聊天会话在 5 秒内从数据库消失。
- **SC-008**：重复邮箱注册请求 100% 被拒绝，不创建任何孤儿账号。
- **SC-009**：第三方邮件服务不可用时，注册流程不会留下未验证的孤儿账号。
- **SC-010**：登录会话在 7 天后自动过期；过期后首个受保护请求必须返回 401。

## Assumptions

- **默认策略**：邮箱密码注册的邮箱**必须**验证后才能登录；OAuth 用户邮箱默认已验证（由 provider 保证）。
- **不实现**：忘记密码 / 密码重置流程（Stage 2 再做）。
- **不实现**：MFA / 手机号 / 短信验证 / 通行密钥。
- **不实现**：账号恢复 / 软删除 / 账号封禁 / 管理员后台。
- **不实现**：团队 / Workspace / Thread 协作者 / 分享。
- **数据处置**：现有 57 个无主 thread 全部丢弃，数据库全量重置；本期不再兼容历史数据。
- **邮件服务商**：业务侧已决定使用 Resend 作为邮件服务商；具体 SDK 选型、模板策略、退订链接等实现细节记录在 plan 阶段。本期使用 **Resend 免费版**（每日配额 100 封）；超出配额时按 FR-025 行为处理（拒绝注册 + 提示稍后重试）。
- **认证框架**：业务侧已决定使用 **Better Auth**（自托管、Drizzle 原生 adapter、内置 OAuth + 邮箱密码 + 邮箱验证）；前后端沿用现有栈（Next.js 16 App Router + Drizzle ORM + assistant-ui）；具体集成代码、schema 映射、SDK 选型记录在 plan 阶段。
- **用户身份最小集**：以 email 为主标识，name/image 可选；不强制收集真实姓名。
- **会话模型**：单一登录态，不维护多设备会话列表展示给用户；登出仅使当前会话失效。固定有效期 **7 天**（用户偏好短于业界常见的 30 天，以降低账号盗用风险窗口）。
- **CSRF / XSS**：依赖框架默认防护；本期不单独实现额外防护层。
- **国际化**：UI 文案英文优先（与现有项目代码/UI 风格一致）；本期不做其他语言翻译。注意：`spec.md` / `plan.md` / `tasks.md` 本身仍是中文（宪法原则六），仅面向最终用户可见的字符串使用英文。
- **文档交付物**（强制包含在本期实现里）：
  - `.env.example` 必须列出所有认证相关环境变量（含占位符与简短说明），不依赖开发者自行推断。
  - `README.md` 必须新增/更新"Authentication"章节，描述注册、登录、登出、OAuth 配置入口、邮件验证流程。
  - `docs/AUTH.md` 必须新建，作为认证功能的完整运维手册：覆盖 GitHub/Google OAuth 应用创建、Resend 域名/发件人配置、本地开发与生产环境变量、常见排错、Stage 2 路线图。
- **邮件模板选型**：优先选用社区成熟的开源 HTML 邮件模板（如 react-email / maizzle），不自行从零设计 HTML/CSS，确保跨邮件客户端兼容性。
